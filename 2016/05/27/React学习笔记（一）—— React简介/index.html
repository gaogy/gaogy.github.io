React学习笔记（一）—— React简介

读书笔记
  • JavaScript
  • React
  • 前端


React 是一个 Facebook 和 Instagram 用来创建用户界面的 JavaScript 库…

web开发早期

在web开发的早期时代（据史书记载，那段不堪回首的岁月被称之为web1.0时代 (╯▽╰)），开发过程十分的简单明快，构建web应用的大致过程就是，浏览器向服务器端发送请求，然后由服务器端响应请求并返回页面内容，基本可以概括为：朕（服务器）给你（浏览器）的是你的，朕不给你的你不能抢。

这种模式下开发者无需关心浏览器端发生了什么，通常5个人以内，不分前后端就可以close掉一个小项目，当然是在业务不太复杂的情况下。然而业务往往是会变复杂滴（个人感觉这和熵增原理存在着某些必然联系(￢_￢)），当业务变得复杂时，很多开心的事就会发僧了，Because往往业务一复杂，人就得多，人一多，花样就多，花样一多，维护的成本就会变大…

还有一个就是用户体验问题，在这种模式下很难打造出极佳的用户体验，因为无论用户想做点什么，都需要向服务器发请求，并等待服务器端的响应，这会导致用户失去在页面上所积累的状态。毕竟我们是服务行业，服务行业的主旨是什么？——客户就是上帝。上帝要是不开心了，你还想不想吃小馒头了。因此为了小馒头，，，Σ( ° △ °|||)︴，，，啊呸，不对，是为了更好的用户体验！！人们开始开发类库。

类库

为了更好的用户体验，人们开始开发类库，使用JavaScript在浏览器端渲染应用。这些类库使用的方法也不尽相同，简单的使用带参数的模板，复杂的就完全掌控整个应用。随着开发者在越来越大的项目中使用这些类库，应用也变得越来越难以把握，因为这些应用是一系列相互作用的结果。

React简介

React 是一个 Facebook 和 Instagram 用来创建用户界面的 JavaScript 库。发源自FaceBook的PHP框架XHP的一个分支，XHP作为一个PHP框架，其主旨在于每次有请求进来时渲染整个页面。React的产生是为了把这种重新渲染整个页面的PHP式工作流带到客户端应用中来。

React本质上是一个“状态机”，可以帮助开发者管理复杂的随着时间而变化的状态。它以一种精简的模型实现了这一点。React只关心两件事：

  • 最小化重绘DOM
  • 单事件处理器

最小化重绘DOM

React不处理Ajax，路由和数据存储，也不规定数据的组织方式。如果非要问它是什么，我们可以把它看做MVC里的“V”。React的精简允许你将其集成到各种各样的系统中。我们几道，在页面渲染过程中，频繁的操作DOM元素，会导严重影响页面性能。而React通过一个虚拟的DOM实现了一个非常强大的渲染系统，在React中可以做到对DOM的最小化重绘（_( ﾟДﾟ)ﾉ江湖传言可以做到只更新不读取…）。

React把视图更新简化为一个render函数，render函数只关心两个参数，分别是配置项和状态。render函数读入当前状态，并返回一个快速的DOM描述。只要React被告知状态有变化，其会重新运行函数，计算出新的DOM描述，然后判断差异后，进行最少的必要更新。

单事件处理器

在web开发过程当中，过多的事件处理器会增大程序的开销，影响应用的性能。通常我们会使用事件委托来减少事件处理器的数量。而React在整个应用中只使用单个的事件处理器，并且会把所有的事件委托到这个事件处理器上，这一点也大大提升了React的性能。
